💡 ¿Qué es CI/CD?

CI/CD es un método que automatiza las etapas de desarrollo de software para entregar aplicaciones de forma rápida y sin errores. Se divide en dos partes principales:

🧪 CI: Integración Continua (Continuous Integration)

Es la fase donde cada vez que un programador sube un cambio al repositorio (Git), un servidor automatizado compila el código y ejecuta pruebas unitarias para confirmar que todo funciona correctamente antes de unirlo al proyecto principal.

🚀 CD: Entrega o Despliegue Continuo (Continuous Delivery / Deployment)

Es la fase que toma el código ya verificado en el CI y lo empaqueta automáticamente para enviarlo a los servidores de producción (la nube), de modo que los usuarios finales puedan usar las nuevas mejoras sin interrupciones.

🚗 Analogía de la vida real: La Fábrica de Autos

Imagina que estás construyendo un coche en una planta de ensamblaje moderna:

🛠️ El proceso sin automatización (El caos antiguo):
Cada mecánico arma una parte del auto por su cuenta. Al final, intentan unir las piezas y descubren que las puertas no encajan y los frenos fallan. Tienen que desarmar todo, probar pieza por pieza manualmente y retrasar la entrega meses.

🤖 El proceso con CI/CD y Docker:
  • CI (La inspección en línea): Conforme se ensambla cada parte del motor, un brazo robótico automático la prueba. Si hay un tornillo suelto, suena una alarma de inmediato (CI fallido) y se corrige al instante.
  • CD (La entrega estandarizada): Cuando el coche pasa todas las pruebas, se monta en un camión de transporte estandarizado (Docker) que lo lleva de forma segura a cualquier concesionario del país sin importar el estado de las carreteras.

🐳 ¿Qué relación tiene con Docker?

Docker y CI/CD son tecnologías independientes, pero van de la mano por tres motivos arquitectónicos principales:

Razón ¿Cómo funciona sin Docker? ¿Cómo mejora con Docker?
1. Consistencia del entorno El servidor de pruebas CI tiene una versión de Node.js v18, pero tu computadora tiene v20. Las pruebas pasan localmente pero fallan en el servidor de CI. Garantía: Tanto las pruebas locales como las del servidor de CI corren dentro del mismo contenedor con Node.js v20.
2. El Artefacto es la Imagen El pipeline de CD sube un archivo zip con código fuente. El servidor debe descomprimirlo, instalar librerías npm en caliente y cruzar los dedos para que no haya errores de red. Inmutable: Se construye la Imagen Docker una sola vez, se prueba en CI y esa misma imagen congelada y probada es la que se despliega. No hay instalación en caliente en producción.
3. Limpieza en los Runners Cada proyecto necesita configuraciones distintas (Java, Python, Postgres). El servidor de CI se llena de software basura y librerías en conflicto. Aislamiento: El servidor de CI solo necesita Docker. Cada tarea de pruebas corre en contenedores separados y aislados sin ensuciar la máquina.

📜 Glosario de Conceptos Clave

Término ¿Qué significa en lenguaje sencillo?
Pipeline (Tubería) La secuencia de pasos automáticos que recorre tu código (Prueba ➔ Compila ➔ Despliega).
Runner (Ejecutor) La computadora/servidor que hace el trabajo físico de correr los comandos del pipeline.
Registry (Registro) La tienda o almacén donde se guardan tus imágenes Docker versión por versión (ej. Docker Hub).
Webhook El "teléfono" que avisa al servidor de CI/CD que acabas de subir código a GitHub para que encienda los robots de prueba.

🛠️ ¿Cómo se configura CI/CD en este Proyecto? (Activo y Real)

¡Este proyecto ya se encuentra 100% configurado para CI/CD! No es un ejemplo simulado. Hemos creado y configurado el archivo de automatización real en la raíz del espacio de trabajo:

Ruta física del archivo: .github/workflows/ci-cd.yml

Paso a Paso de nuestro Pipeline configurado:

  1. Fase 1: Descarga y Entorno: GitHub Actions descarga el código en sus servidores en la nube y configura Node.js LTS v20 con caché inteligente de NPM.
  2. Fase 2: Testeo del Backend: Se instalan las librerías limpiamente (`npm ci`) y se ejecuta el comando de pruebas automatizado (`npm test`).
  3. Fase 3: Compilación del Frontend (Angular): Se descargan las dependencias del frontend y se genera el compilado estático de producción de Angular.
  4. Fase 4: Validación de Docker: Se ejecuta `docker compose build` en caliente en la nube para asegurar que los Dockerfiles del Backend (Node) y Frontend (Nginx) se compilan perfectamente.

Receta real activa de nuestro proyecto (ci-cd.yml):

name: Pipeline de Integracion y Despliegue Continuo (CI/CD)

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    name: Integracion Continua (CI)
    runs-on: ubuntu-latest

    steps:
      - name: Descargar codigo del repositorio
        uses: actions/checkout@v4

      - name: Configurar Node.js (LTS)
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
          cache-dependency-path: |
            frontend/package-lock.json
            backend/package-lock.json

      - name: Instalar dependencias del Backend
        run: |
          cd backend
          npm ci

      - name: Ejecutar pruebas del Backend
        run: |
          cd backend
          npm test

      - name: Instalar dependencias del Frontend
        run: |
          cd frontend
          npm ci --legacy-peer-deps

      - name: Compilar Frontend (Angular)
        run: |
          cd frontend
          npm run build -- --configuration=production

      - name: Validar compilacion de Docker Compose
        run: |
          docker compose build
