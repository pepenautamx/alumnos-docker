🗄️ Fase 1: Creación de la Base de Datos (PostgreSQL en Docker)

El primer paso fue levantar el motor de base de datos de manera aislada y crear la estructura física de la tabla de calificaciones de manera manual o automatizada.

1. Comando para inicializar el contenedor temporal de desarrollo:

docker run --name db_alumnos -e POSTGRES_USER=pepe_admin -e POSTGRES_PASSWORD=supersecretpassword -e POSTGRES_DB=escuela_db -p 5435:5432 -d postgres:15-alpine
Explicación de las banderas del comando:
Parámetro ¿Qué significa para un novato?
docker run Ordena a Docker crear y encender un contenedor nuevo a partir de una imagen.
--name db_alumnos El nombre que le asignamos al contenedor para poder controlarlo, apagarlo o ver sus logs fácilmente.
-e POSTGRES_USER/... Inyecta variables de entorno (parámetros de configuración) para definir el usuario, contraseña y base de datos iniciales.
-p 5435:5432 Mapea los puertos. Abre la base de datos en el puerto 5435 de tu computadora (externo) y la redirecciona al 5432 (puerto interno de Postgres en el contenedor).
-d Modo "detached". Indica que el contenedor debe correr en segundo plano, liberando tu terminal inmediatamente.
postgres:15-alpine La "receta" oficial. Indica que descargue la versión 15 de Postgres con el sistema ligero Alpine Linux (5MB).

require('dotenv').config(); // Carga de variables de entorno

const { ApolloServer } = require('@apollo/server');
const { startStandaloneServer } = require('@apollo/server/standalone');
const { Pool } = require('pg');
const cors = require('cors');

// Configuración de la conexión a Postgres
const pool = new Pool({
  host: process.env.DB_HOST || 'localhost',
  port: parseInt(process.env.DB_PORT || '5435', 10),
  user: process.env.DB_USER || 'pepe_admin',
  password: process.env.DB_PASSWORD || 'supersecretpassword',
  database: process.env.DB_DATABASE || 'escuela_db',
});

// Esquema de GraphQL
const typeDefs = `#graphql
  type Alumno {
    id: ID!
    nombre: String!
    apellidos: String!
    matematicas: Float
    espanol: Float
    historia: Float
  }

  type Query {
    obtenerAlumnos: [Alumno]
  }
`;

// Resolvers encargados de ir por los datos reales a la base de datos
const resolvers = {
  Query: {
    obtenerAlumnos: async () => {
      try {
        const res = await pool.query('SELECT * FROM alumnos ORDER BY id ASC');
        return res.rows;
      } catch (err) {
        console.error("Error consultando la base de datos", err);
        throw new Error("No se pudieron obtener los datos");
      }
    },
  },
};

// Levantamos el servidor Apollo aplicando middleware de CORS
const server = new ApolloServer({ typeDefs, resolvers });

(async () => {
  const { url } = await startStandaloneServer(server, {
    listen: { port: 4000 },
    // Permitir accesos del puerto 4200 de Angular
    cors: {
      origin: '*',
      credentials: true
    }
  });
  console.log(`🚀 Servidor GraphQL listo en: ${url}`);
})();

3. Creación y código del Servicio de Datos (frontend/src/app/services/alumnos.service.ts):

import { Injectable, inject } from '@angular/core';
import { Apollo, gql } from 'apollo-angular';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';

@Injectable({
  providedIn: 'root'
})
export class AlumnosService {
  private apollo = inject(Apollo);

  obtenerAlumnos(): Observable<any[]> {
    return this.apollo.query<any>({
      query: gql`
        query ObtenerAlumnos {
          obtenerAlumnos {
            id
            nombre
            apellidos
            matematicas
            espanol
            historia
          }
        }
      `
    }).pipe(
      map(resultado => resultado.data.obtenerAlumnos)
    );
  }
}

📦 Fase 6: Dockerización Completa del Frontend (Nginx en Producción)

Para lograr portabilidad total del frontend, evitamos usar la consola de desarrollo local (`ng serve`) y empaquetamos el compilado estático de Angular dentro de un servidor Nginx ligero.

1. Crear el Dockerfile en /frontend/Dockerfile:

# Etapa 1: Compilación de la aplicación de Angular
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build -- --configuration=production

# Etapa 2: Servidor Web Nginx ligero para producción
FROM nginx:alpine
COPY --from=build /app/dist/frontend/browser /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
Explicación detallada del Dockerfile multi-etapa:
Instrucción ¿Qué hace en esta construcción multi-etapa?
FROM node:20-alpine AS build Inicializa la **Etapa 1 (Compilación)**. Descarga Node.js y etiqueta esta fase como build para poder referenciarla más adelante.
RUN npm run build -- ... Ejecuta la compilación de Angular en modo producción, generando archivos estáticos optimizados (.js, .css, .html) en el directorio dist/frontend/browser.
FROM nginx:alpine Inicializa la **Etapa 2 (Ejecución)**. Descarga un servidor web **Nginx** extremadamente ligero y descarta todo lo anterior (Node.js y código fuente), dejando una imagen final muy pequeña.
COPY --from=build ... Copia los archivos estáticos generados en la Etapa 1 (--from=build) directamente al directorio de Nginx para que puedan ser mostrados al público.
EXPOSE 80 Informa al contenedor que escuche en el puerto estándar HTTP (puerto 80).
CMD ["nginx", "-g", "daemon off;"] Arranca el servidor Nginx en primer plano para mantener el contenedor encendido sirviendo la web.

🤖 Fase 7: Automatización del Pipeline de CI/CD (GitHub Actions)

Configuramos un pipeline automático que valida cada cambio al hacer push a la rama principal (main), garantizando que todo compile correctamente antes del despliegue.

1. Crear el archivo del Workflow en .github/workflows/ci-cd.yml:

name: Pipeline de Integracion y Despliegue Continuo (CI/CD)

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    name: Integracion Continua (CI)
    runs-on: ubuntu-latest

    steps:
      - name: Descargar codigo del repositorio
        uses: actions/checkout@v4

      - name: Configurar Node.js (LTS)
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
          cache-dependency-path: |
            frontend/package-lock.json
            backend/package-lock.json

      - name: Instalar dependencias del Backend
        run: |
          cd backend
          npm ci

      - name: Ejecutar pruebas del Backend
        run: |
          cd backend
          npm test

      - name: Instalar dependencias del Frontend
        run: |
          cd frontend
          npm ci --legacy-peer-deps

      - name: Compilar Frontend (Angular)
        run: |
          cd frontend
          npm run build -- --configuration=production

      - name: Validar compilacion de Docker Compose
        run: |
          docker compose build

2. Explicación detallada de nuestro Workflow:

Bloque / Comando ¿Qué hace en el Pipeline?
on: push: branches: [main] Define los disparadores (triggers). El pipeline se ejecuta automáticamente cada vez que se sube código (push) o se abre un pull request a la rama main.

🛠️ Fase 5: Resolución de Errores en Producción (Troubleshooting)

El valor real de este desarrollo radicó en cómo solucionamos las incompatibilidades del ecosistema moderno:

1. El Bug de Criptografía ("crypto is not defined")

Las herramientas de Apollo Server requerían APIs globales de cifrado ausentes por defecto en Node 18 Alpine. Lo solucionamos actualizando el entorno base de nuestro Dockerfile a la versión LTS más robusta: FROM node:20-alpine.

2. Bloqueo de Seguridad entre Puertos (CORS)

Al intentar leer la API desde el puerto 4200 de Angular hacia el puerto 4000 de Node, el navegador bloqueaba las respuestas. Solucionamos el problema reconfigurando el inicio del backend en index.js inyectando middleware de control de accesos explícito: cors: { origin: '*' }.

3. Conflicto de Versiones en el Árbol de Dependencias de NPM

El instalador de paquetes arrojaba el error ERESOLVE unable to resolve dependency tree debido a discrepancias estrictas entre Angular 19 y los paquetes de Apollo. Forzamos la instalación segura mitigando la restricción visual mediante el flag: --legacy-peer-deps.

4. Conflicto del Directorio Raíz de TypeScript

El compilador de TS marcaba error en tsconfig.json al no identificar con exactitud el origen de la compilación. Lo solucionamos fijando de forma explícita el parámetro "rootDir": "." en las opciones del compilador.

5. Choque de Sintaxis de Bloques en Angular Moderno ("Incomplete block apollo")

El nuevo motor de plantillas de Angular confundía el carácter @ de las dependencias de npm con directivas del framework. Lo blindamos usando su correspondiente entidad de escape HTML: &#64; o su equivalente seguro.
