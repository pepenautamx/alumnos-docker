💡 ¿Qué es Docker?

Docker es una plataforma que permite empaquetar una aplicación junto con todo lo que necesita para funcionar (librerías, configuraciones, bases de datos) dentro de una unidad llamada contenedor. Esto garantiza que el software funcione exactamente igual en tu computadora, en la de un compañero, o en un servidor en la nube.

📦 Ejemplo de la vida real: El Contenedor Marítimo
Antes de 1956, transportar mercancías en barcos era un caos: las cajas sueltas se mezclaban e incluso dañaban entre sí. La solución fue el contenedor estandarizado. No importa qué contenga; el contenedor mide lo mismo, encaja perfectamente, se cierra herméticamente y cualquier grúa del mundo lo puede mover de forma uniforme.

¡Docker hace exactamente lo mismo con el software!

🛠️ El Dockerfile de nuestro Backend (Explicado línea por línea)

Este es el archivo que creamos dentro de la carpeta /backend para indicarle a Docker cómo debe empaquetar nuestra API de GraphQL:

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 4000
CMD ["node", "index.js"]

🔎 ¿Qué hace cada instrucción?

Instrucción Explicación Técnica Destripada
FROM node:20-alpine Define el sistema operativo base. En lugar de instalar un Linux gigante, bajamos Alpine (distribución ligera de 5MB) que ya viene con Node.js versión 20 preinstalado.
WORKDIR /app Crea una carpeta llamada /app dentro del contenedor y se para en ella para el directorio de trabajo virtual.
COPY package*.json ./ Copia los archivos de dependencias de tu computadora al contenedor aprovechando la caché de capas de Docker.
RUN npm install Ejecuta el comando para descargar todas las librerías necesarias (como apollo-server, graphql, y pg) directo al contenedor.
COPY . . Copia todo el resto del código fuente del backend (como tu archivo index.js) desde tu computadora al contenedor.
EXPOSE 4000 Informa que el contenedor escuchará peticiones en el puerto 4000 de forma interna.
CMD ["node", "index.js"] El comando final que se ejecuta únicamente cuando el contenedor se enciende para mantener el servidor web arriba.

🎼 El docker-compose.yml (Explicado línea por línea)

Este archivo vive en la raíz del proyecto y sirve para levantar múltiples contenedores conectados entre sí:

version: '3.8'

services:
  db:
    image: postgres:15-alpine
    container_name: db_alumnos
    restart: always
    environment:
      POSTGRES_USER: pepe_admin
      POSTGRES_PASSWORD: supersecretpassword
      POSTGRES_DB: escuela_db
    ports:
      - "5435:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./db-init:/docker-entrypoint-initdb.d

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: api_alumnos
    restart: always
    ports:
      - "4000:4000"
    environment:
      DB_HOST: db
      DB_PORT: 5432
      DB_USER: pepe_admin
      DB_PASSWORD: supersecretpassword
      DB_DATABASE: escuela_db
    depends_on:
      - db

volumes:
  postgres_data:

🔎 ¿Qué hace cada bloque y parámetro?

Parámetro Explicación Técnica Destripada
version: '3.8' Especifica la versión del motor de Docker Compose que vamos a usar para el archivo.
services: Indica el inicio de la definición de los contenedores que van a trabajar en equipo (db y backend).
db -> image Le dice a Docker que descargue la imagen oficial ya optimizada de PostgreSQL 15 desde Docker Hub.
container_name: ... Asigna un nombre fijo al contenedor (ej. db_alumnos y api_alumnos) para identificarlos fácilmente.
restart: always Si el contenedor se cae o la máquina se reinicia, Docker volverá a encender el servicio automáticamente.
environment: Inyecta variables de entorno para configurar las credenciales de acceso de las bases de datos y la API.
ports: Redirección de puertos. El Postgres corre interno en el 5432 pero lo mapeamos al 5435 afuera para no generar conflictos en tu máquina.
volumes: Mapea directorios. En este compose usamos dos tipos:
  • Volumen Nombrado (postgres_data): Docker maneja los archivos en su propio espacio interno para persistir los datos de la BD incluso si destruyes el contenedor.
  • Bind Mount (./db-init:...): Enlaza de forma directa una carpeta real de tu computadora al contenedor (en este caso para inicializar scripts SQL).
backend -> build: Le ordena a Docker Compose entrar a ./backend y compilar la imagen usando el Dockerfile local.
depends_on: - db Establece el orden de encendido obligando al backend a esperar a que Postgres esté 100% levantado primero.

📜 Chuleta de Comandos Básicos (Cheat Sheet)

Los comandos indispensables para operar este ecosistema en el día a día desde tu terminal:

Comando ¿Para qué sirve?
docker compose up -d --build Compila las imágenes locales, crea las redes, levanta los contenedores en segundo plano (-d) e inicializa los servicios.
docker compose ps Lista todos los contenedores activos de tu proyecto actual indicando su estado y puertos.
docker compose logs -f Muestra en tiempo real las salidas de terminal (logs) de todos los contenedores para depurar errores.
docker compose down Apaga los contenedores y destruye las redes virtuales creadas (los datos no se pierden gracias al volumen mapeado).
docker exec -it api_alumnos sh Abre una terminal interactiva (-it) usando shell (sh) dentro del contenedor en ejecución del backend.
docker system prune -a --volumes Comando de limpieza profunda. Elimina todas las imágenes, contenedores apagados y redes huérfanas para liberar espacio en disco.
